# Benchmark methodology

Status: Phase 11 planning surface. No benchmark result claim is made by this document.

## Goal

This methodology defines how to compare ShyftR with other memory systems, including mem0, using reproducible local scripts and public-safe reports.

The benchmark must answer two different questions:

1. How well does each memory backend retrieve and support answers?
2. What control, audit, and review qualities does each backend preserve while doing so?

ShyftR should be evaluated as a memory backend with local cells, reviewed memory, packs, provenance, and feedback. Other systems should be evaluated through equivalent adapters where possible.

## Comparator systems

Initial comparators:

- ShyftR local cell backend;
- no-memory baseline;
- mem0 OSS backend;
- mem0 Cloud backend when credentials are explicitly supplied;
- simple local BM25 or vector baseline after the first adapter harness is stable.

A comparator may be marked `skipped` when its dependency or credentials are missing. Skips must appear in the report.

## Dataset order

Start small and expand only after the harness is stable:

1. synthetic mini fixture committed or generated by the repo;
2. LOCOMO mini fixture;
3. LOCOMO standard run;
4. LongMemEval subset;
5. BEAM 100K;
6. larger BEAM buckets only after timeout, cost, and resume controls are proven.

Large third-party datasets should be downloaded by script or referenced by documented setup. They should not be vendored into the repo by default.

## Common run shape

Every backend should follow the same runner shape:

```text
ingest -> search -> runner answer -> runner judge/evaluate -> report
```

For the first public-safe run, answer generation should be runner-owned rather than backend-owned. This keeps the comparison focused on memory behavior instead of agent-loop differences.

## Required report metadata

Every report must include:

- schema version;
- run id;
- generated timestamp;
- ShyftR git SHA when run from this repo;
- dataset name and version;
- backend names and config summaries;
- answerer model;
- judge model;
- embedding model where applicable;
- top-k values;
- ingest, search, answer, and judge timing;
- token usage and cost estimate method;
- skipped backends and reasons;
- limitations;
- claims allowed;
- claims not allowed.

## Retrieval metrics

Report retrieval quality at fixed top-k values. For multi-cutoff runs, query each backend once at the largest requested k and compute the smaller cutoffs from that same ranked list:

- recall at k;
- precision at k;
- mean reciprocal rank;
- nDCG;
- answer-support coverage;
- stale or conflicting-memory retrieval rate.

## Answer metrics

When answer generation and judging are enabled, report:

- correctness against dataset labels;
- fixed judge score;
- temporal correctness;
- multi-hop correctness;
- unknown or abstention behavior;
- failure bucket counts.

The answerer and judge models must be fixed and disclosed for a run.

## Control and audit metrics

ShyftR's benchmark track must also report control qualities that retrieval-only scores do not capture:

- provenance coverage;
- review-gate compliance;
- replay status;
- sensitivity leak rate;
- append-only preservation;
- feedback visibility;
- pack compactness;
- useful context per token.

Comparator systems may report `not_supported` for a control metric when no equivalent exists. That is useful information and should not be hidden.

## Fairness rules

A benchmark run must keep these fixed across comparable backends:

- dataset split;
- question order;
- top-k cutoffs;
- answerer model;
- judge model;
- timeout and retry policy;
- resume policy for interrupted or repeated runs;
- token accounting method;
- cold or warm run mode.

If embedding models differ because a backend requires its own pipeline, the report must disclose that difference.

## Claim rules

Allowed claims after a reproducible run:

- backend A scored X on metric Y under dataset Z and config C;
- backend A had lower or higher latency under the pinned run config;
- ShyftR preserved provenance/review/replay fields for N of M returned items under the run config.

Not allowed without further evidence:

- broad superiority claims;
- hosted or production claims;
- task-success lift claims not measured by the run;
- claims based on private data that cannot be reproduced;
- claims that ignore missing comparator dependencies or skipped backends.

## First implementation target

The first implementation target is a tiny fixture-safe adapter harness that compares ShyftR with a no-memory baseline and emits report JSON. mem0 OSS compatibility comes after that contract is stable.
