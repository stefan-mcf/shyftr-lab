# ShyftR Naming Conventions

> Source: operator-local concept document, sanitized into this public note.
> Converted from PDF to Markdown for repository reference. Source PDF pages: 4.

This document records the current naming recommendations for the major concepts in ShyftR. The goal of this scheme is to use terms that are memorable, easy to teach and understand, and consistent with the project’s "battery/memory cell" theme. Where previous implementations used different names, a mapping is provided to aid migration.

## Guiding Principles

- Clarity over cleverness – Names should immediately convey their role in the lifecycle. Readers

should not need to understand the entire system before they can reason about a given term.

- One-to-one mapping – Each stage of the memory lifecycle should have exactly one canonical

name. Avoid synonyms that could cause confusion.

- Theme coherence – The electrical storage metaphor (cells, memories, circuits) is used consistently

to hint at how information moves through the system.

## Canonical Terms

The following table lists the official vocabulary with a brief definition of each. Later sections provide more detailed explanations. Concept Definition cell A bounded memory store. Each cell encapsulates its own ledger files and indexes and can be mounted on an agent, a project, a user or a sector. evidence Raw evidence entering the system, such as logs, task closeouts or documents. evidences are captured verbatim in append-only ledgers. candidate A candidate memory extracted from a evidence. candidates are proposed statements or heuristics that require review before being promoted. memory An approved, durable memory. memories are the authoritative facts, rules or heuristics used to build packs for tasks. Circuit A distilled pattern or theme derived from multiple related memories. Circuits represent reusable workflows or multi-step heuristics. rule A high-authority rule or rule promoted from repeated circuits. rules are shared across cells and act as guardrules or policies. grid The retrieval layer responsible for searching memories (e.g. sparse FTS, vector index). grids are rebuildable and never the source of truth. pack A bounded, task-ready context compiled from memories and circuits. packs include guidance, caution items, verification heuristics and conflicts. Concept Definition feedback feedback feedback after a pack is used. feedbacks record whether retrieved memory was useful, harmful, ignored or missing. regulator The policy layer that enforces boundaries on ingestion, review and promotion. The regulator defines what may be automatically promoted or requires human review.

### Detailed Explanations

cell A cell is the fundamental container for all memory operations. It holds append-only ledger files (evidences.jsonl, candidates.jsonl, memories/approved.jsonl, etc.) as well as indexes under grid/. A cell can be mounted onto any layer of an agent system – such as a user profile, project, runtime sector or domain – allowing the same code to operate in different contexts. Multiple cells can be connected later through federation, but the cell remains the unit of authority and quarantine. evidence (raw evidence) A evidence captures raw evidence that might be important later. It could be a task closeout, a log line, a document, a chat message or a human note. evidences are stored verbatim in append-only JSONL files and are never modified. evidences do not influence retrieval directly – they must first be analysed to produce candidates. candidate (candidate memory) A candidate is a potential piece of knowledge extracted from a evidence. candidates include the proposed statement or heuristic and metadata such as source hash, tags and a suggested kind (e.g. guidance, caution, verification). candidates live in a pending state until they are explicitly reviewed. The review can approve the candidate, reject it, split or merge it into multiple candidates, or leave it pending. memory (approved memory) A memory is what happens when a candidate is approved. memories are durable and authoritative. They are stored in memories/approved.jsonl and become candidates for retrieval. memories have confidence scores that evolve over time via feedbacks and sweep proposals, and they include provenance back to the original evidence and review events. Circuit (distilled pattern) A circuit represents a recurring pattern, workflow or multi-step heuristic distilled from several memories. For example, multiple memories about testing before deploying might form a single circuit representing the “test-then-deploy” process. Circuits are optional in the MVP, but the term is reserved for this distilled intermediate layer. (Previously this concept was called a “pattern”; it was renamed because “circuit” more clearly conveys connected flow and reuse.) rule (promoted rule) A rule is a high-authority rule or policy promoted from circuits. rules act as guardrules across cells and define global constraints that should be followed by agents or runtimes. Because rules can affect many cells, they require strict review and careful promotion. grid (retrieval layer) The grid is responsible for efficient retrieval of memories. Different grid implementations might include sparse full-text search, vector embeddings, hybrid scoring or other indexing strategies. The grid never stores authoritative data – it can always be rebuilt from the canonical ledgers. grid status includes metadata such as the embedding model, ledger offsets and whether the index is stale. pack (task-ready context) A pack is the context assembled for a specific task. It contains selected memories, circuits and rules relevant to the query, organised into sections such as guidance, caution items, verification heuristics and conflict warnings. packs include provenance and scoring information so that agents can memory why a particular memory was selected or suppressed. packs are bounded to meet token budgets and can be generated in different modes (e.g. conservative, balanced, exploratory). feedback (feedback feedback) A feedback records what happened after a pack was used. feedbacks capture whether retrieved memory was useful, harmful, ignored, contradicted, over-retrieved or missing. This feedback is appended into feedback ledgers and drives learning through confidence events and retrieval affinity adjustments. feedbacks are essential for making the memory self-learning and adaptive. regulator (policy layer) The regulator enforces boundaries at every stage of the lifecycle. It determines what evidences are allowed to be ingested, what candidates are acceptable to extract, when a candidate can be auto-promoted to a memory, and how retrieval and promotion policies adapt based on feedback. The regulator enables safe automation by preventing unreviewed content from silently becoming durable memory.

## Rationale for Renaming “pattern” to “Circuit”

The prototype codebase uses the term pattern or pattern to refer to an intermediate layer between memories and rules. However, the name “pattern” proved opaque – users often could not tell what it meant. The new term Circuit retains the electrical metaphor while better conveying that it is a pattern of connected memories that can be reused. A circuit implies a flow or loop of energy, just as a memory circuit represents a recurring pattern of knowledge.

## Summary

The above conventions strike a balance between creativity and clarity. By adopting a clear, themed vocabulary, ShyftR can communicate its concepts to developers, operators and researchers more effectively. The canonical sequence of the memory lifecycle is now: evidence → candidate → memory → Circuit → rule ↓ pack → feedback Where the regulator governs every transition and the grid facilitates retrieval without becoming an authority. These names make the project’s architecture easier to explain and implement, and they provide a stable foundation for future documentation and API design.
